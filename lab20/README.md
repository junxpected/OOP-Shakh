# Лабораторна робота №20  
## Тема: SRP — декомпозиція OrderProcessor

### Мета роботи
Метою цієї лабораторної роботи є застосування принципу **Single Responsibility Principle (SRP)**  
шляхом рефакторингу класу `OrderProcessor`, який спочатку виконує декілька різних задач.


### Коротко про SRP
SRP означає, що **клас повинен мати лише одну відповідальність**  
і лише одну причину для зміни.

Якщо клас одночасно:
- перевіряє дані
- працює з базою
- відправляє email

— це порушує SRP.

### Опис завдання
У роботі реалізовано два етапи:

1. **Початковий варіант (порушує SRP)**  
   Клас `OrderProcessor`:
   - перевіряє замовлення
   - зберігає його
   - відправляє email
   - змінює статус

2. **Рефакторинг з дотриманням SRP**  
   Логіка була розділена на окремі компоненти:
   - `IOrderValidator` — валідація замовлення
   - `IOrderRepository` — збереження замовлення
   - `IEmailService` — відправка email
   - `OrderService` — координує процес обробки

Залежності передаються через конструктор (Dependency Injection).


### Демонстрація роботи
У методі `Main`:
- створюється валідне замовлення
- створюється невалідне замовлення
- викликається `ProcessOrder`
- результати роботи виводяться в консоль


### Висновок
Після застосування SRP:
- код став зрозумілішим
- кожен клас відповідає лише за одну задачу
- програму легше підтримувати та розширювати

Принцип SRP допомагає уникати «перевантажених» класів  
і робить архітектуру більш гнучкою.
