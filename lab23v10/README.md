# Лабораторна робота №23

**Тема :** Виявлення порушень принципів ISP (Interface Segregation Principle) та DIP (Dependency Inversion Principle) і їх усунення за допомогою рефакторингу.

---

**1)** Аналіз коду до рефакторингу

**Початкова структура**

Було створено клас `HeroAction` , який виконує три дії:

 - `Attack()`

 - `Heal()`

 - `Talk()`

Він напряму створював свої залежності:

```csharp
public HeroAction()
{
    weaponSystem = new WeaponSystem();
    medicalKit = new MedicalKit();
    dialogueManager = new DialogueManager();
} 
```


**Порушення DIP**

Принцип DIP такий:

Модулі верхнього рівня не повинні залежати від модулів нижнього рівня.
Вони повинні залежати від абстракцій.

У початковій версії:

`HeroAction` залежить від конкретних класів:

`WeaponSystem`

`MedicalKit`

`DialogueManager`

Об'єкти створюються всередині класу і створюється жорсткий зв’язок.

Недоліки:

- Неможливо легко замінити реалізацію

- Важко тестувати

- Порушується гнучкість системи


Порушення ISP

Принцип ISP говорить: Клієнт не повинен залежати від методів, які він не використовує.

У початковій версії:

Герой завжди залежить від лікування і діалогів. Навіть якщо персонаж тільки атакує — він все одно має залежності від інших систем. Це означає, що клас має зайві залежності.

**2)** Рефакторинг

Було виконано два основних кроки:

 1. Виділення вузьких інтерфейсів (ISP)

Створено окремі інтерфейси:
```csharp
public interface IAttack
{
    void Attack();
}
public interface IHeal
{
    void Heal();
}
public interface ITalk
{
    void Talk();
}
```
Тепер кожна система реалізує тільки свій інтерфейс.

 **2.** Впровадження залежностей - DIP

Замість створення об’єктів всередині класу, залежності передаються через конструктор:
```csharp
public HeroAction(IAttack attack, IHeal heal, ITalk talk)
{
    this.attack = attack;
    this.heal = heal;
    this.talk = talk;
}
```
У `Program.cs` відбувається конфігурація:
```csharp
var weapon = new WeaponSystem();
var medKit = new MedicalKit();
var dialogue = new DialogueManager();

var hero = new HeroAction(weapon, medKit, dialogue);
```
Тепер `HeroAction` залежить від інтерфейсів, а не від конкретних класів.

**3)** Опис змін

Було виконано:

- Виділено три окремі інтерфейси

- Класи реалізують тільки потрібний функціонал

- Видалено створення залежностей всередині HeroAction

- Додано Dependency Injection через конструктор

 Результат після рефакторингу

✔ Дотримується ISP
✔ Дотримується DIP
✔ Система стала гнучкішою
✔ Код легше тестувати
✔ Можна легко замінити реалізацію (наприклад, іншу зброю)

# Висновки

Виявлено порушення принципів ISP та DIP

- Проаналізовано проблеми жорсткого зв’язку

- Виконано рефакторинг із використанням інтерфейсів

- Реалізовано Dependency Injection

- Застосування принципів SOLID дозволяє:

- підвищити масштабованість системи
